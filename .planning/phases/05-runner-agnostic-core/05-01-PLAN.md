<objective>
Lift selfmend's internal page-wrapping seam into a public, runner-agnostic core. Two seams become pluggable on `HealContext`: identity (today `testFile`/`testTitle` from `testInfo`) becomes a caller-supplied `scope()` source feeding the `(suite, test)` key; heal-event transport (today hardcoded `attachHealEvent/attachRefusedEvent(ctx.testInfo,...)`) becomes a pluggable `emit(event: SelfmendEvent)`. On top of that seam, expose a public `wrapPage(page, { store, config?, onHeal?, scope? })` returning the BARE wrapped page, plus a sibling `resetScope(page)`.

Purpose: This is the WRAP-01/02/03 surface and the seam Plan 02's fixture refactor sits on. The healing engine (fingerprint, score, decide, rebind) does NOT change; this generalizes the wiring around it.

Output: A refactored `HealContext` (emit + scope source, no testInfo), the new `src/integration/wrap-page.ts` core (wrapPage + resetScope + scope-lifetime + config merge), updated public exports, and test-first coverage of the occurrence-counter lifetime, scope-keyed identity, and config merge.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller -> wrapPage opts | The adopter supplies `store`, `config`, `onHeal`, and `scope()` — none can be trusted to be well-behaved. |
| heal path -> emit/onHeal | Caller code runs inside the heal path via emit -> onHeal; it must never affect the run. |
| caller config -> configSchema | User config crosses a validation boundary (existing T-01-01). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | `emit` / `onHeal` masking a real failure | mitigate | emit is best-effort: the heal/refused path wraps it in try/catch and re-throws the ORIGINAL error regardless (D-08); onHeal is fire-and-forget with throws and rejected promises swallowed (D-07). A failing emit can never suppress, slow, or stall a test. Unit-tested (Task 2 throwing-emit case). |
| T-05-02 | Denial of Service | a throwing `scope()` crashing the page wrap | mitigate | the scope-lifetime controller treats a throwing scope() as fail-safe (fall back to the coarse default tuple) rather than letting it crash the locator factory; never-false-green still holds because a coarse/wrong key is a MISSED heal, not a wrong heal (D-04/D-11). Raw-mode integration control is in Plan 02. |
| T-05-03 | Tampering | caller-supplied `config` partial out of range | mitigate | config merge resolves through `configSchema`, rejecting out-of-range values with readable errors (existing ASVS V5 input validation); no silent coercion. Unit-tested (Task 1 config-merge). |
| T-05-04 | Spoofing | a wrong/missing identity key producing a false green | accept-by-design | the never-false-green guarantee lives in the pure `decide()` (floor + second-best margin + fail-safe on a missing key); a coarse/wrong/absent key yields a MISSED heal (no/ambiguous candidate -> original error re-throws), never a wrong heal. Invariant unchanged; control-tested in raw mode in Plan 02 (D-11). |
| T-05-SC | Tampering | npm/pip/cargo installs | n/a | No new runtime dependencies introduced this plan (refactor only; zod/picocolors already audited). No install tasks. |
</threat_model>
