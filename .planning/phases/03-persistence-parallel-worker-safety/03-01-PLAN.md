<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk file -> in-process store | An on-disk baseline/shard (possibly stale, hand-edited, or written by a different selfmend version) is untrusted input parsed into runtime state. |
| in-process store -> committed file | The serialized text becomes a git-committed, human-readable artifact (must stay PII-free). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering / DoS | parseBaseline / parseShard on an old or malformed file | mitigate | zod `safeParse`; on any failure return the canonical EMPTY store, never throw (D-10, Pitfall 5). Version is `z.literal`, so a future/older format is rejected to empty, never half-read. Proven by schema.test.ts. |
| T-03-02 | Information Disclosure | fingerprint schema / serializer output | mitigate | Schema is STRICT and limited to the eight derived signals; an object carrying `innerHTML`/`outerHTML`/`html` is rejected, so no raw DOM can enter the committed file (D-02, ASVS V8). Asserted in schema.test.ts and serialize.test.ts. |
| T-03-03 | Tampering (integrity) | non-deterministic serialization causing diff churn | mitigate | Deterministic stable sort of entry keys + fingerprint field order + attrs keys; byte-identical output for logically-equal stores (D-03). Proven by serialize.test.ts. |
</threat_model>
