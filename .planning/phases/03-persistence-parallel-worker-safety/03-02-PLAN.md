<objective>
Land the durable persistence seam and the cross-run-stable identity key — the two prerequisites the Reporter merge (Plan 03) wires together. This plan (1) adds the fs adapter `persistence.ts` (the ONLY file that imports `node:fs`), with paths resolved under `rootDir`, a Windows-safe atomic write (temp + rename + EPERM/EBUSY retry), per-worker shard read/write, and shard cleanup; and (2) swaps the Phase 1 run-order `step` key for the occurrence-based key (D-04/D-05) across `store.ts`, `locator-proxy.ts`, and `fixture.ts`, proving the existing heal/no-false-green/ambiguous behavior survives.

Purpose: The committed `baseline.json` is only useful if (a) it can be written/read without corruption (CAP-02/CAP-03 mechanics) and (b) the key that addresses a fingerprint is the SAME on the run that captured it and the run that heals against it. The Phase 1 `step` counter shifts every run and cannot survive — replacing it is the load-bearing change.

Output: `src/store/persistence.ts` (+ temp-dir unit tests), an occurrence-keyed `BaselineStore` with shard serialization + seen-key tracking, an occurrence-key derivation in the proxy/fixture, and updated regression tests proving the swap preserves behavior.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| configured store path -> filesystem | The store dir is derived from rootDir (and an optional test override); a path that escapes rootDir could read/write outside the project. |
| temp file -> committed target | An interrupted/killed run mid-write must not corrupt the readable committed file. |
| heal-time key derivation -> stored baseline | If the heal-time key diverges from the capture-time key, a heal could match the wrong fingerprint or silently fail (D-07 keeps it fail-safe). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering (integrity) | atomicWrite of baseline.json | mitigate | temp-file-in-same-dir + `fs.rename` (atomic on one volume) + EPERM/EBUSY retry; on exhaustion remove temp and throw, so a reader never sees a partial file (RESEARCH Pattern 3, Pitfall 1). Proven by persistence.test.ts. |
| T-03-05 | Tampering | store/shard path resolution | mitigate | All paths via `path.resolve(rootDir, ...)`; the optional test/env override is resolved under rootDir too; a containment assertion proves no traversal escape (Security Domain). |
| T-03-06 | DoS | loadBaseline / readShards on a bad/old/foreign file | mitigate | Delegate to Plan 01 safe loaders (parseBaseline/parseShard); JSON.parse in try/catch; bad files skipped/treated empty, never crash the run (Pitfall 5). |
| T-03-07 | Spoofing / false-green | occurrence key divergence capture-vs-heal | mitigate | Occurrence counter incremented at wrapLocator CREATION (not resolution), per-(content) and per-test, so the index is identical on green and broken runs; a missing key still re-throws (D-05/D-07, Pitfall 4). Proven by occurrence.test.ts. |
</threat_model>
