<objective>
Wire the durable, parallel-safe store together as one vertical: workers load the committed `baseline.json` read-only at worker start and flush their captures + seen-keys to a `parallelIndex`-named shard on worker teardown (CAP-03, D-11); the existing summary Reporter is EXTENDED so `onBegin` records the run-completeness signal and `onEnd` merges all shards, refreshes the baseline, conditionally prunes (opt-in + complete-run only), atomically writes the single committed `baseline.json`, and deletes the shards (D-08/D-09/D-11/D-12). Reconcile `.gitignore` so `baseline.json` is committed but shards/temp stay ignored (D-01/D-12). Prove the two phase requirements with real Playwright specs: a cross-run persist-then-heal (CAP-02) and a `workers>1` concurrent merge (CAP-03), plus a `--grep` run that refreshes-but-does-not-prune (D-09).

Purpose: This is the user-visible payoff of Phase 3 — after this plan a captured fingerprint survives across runs and a parallel suite produces one clean, diffable committed baseline. It depends on the pure layer (Plan 01) and the persistence adapter + occurrence key (Plan 02) already proven.

Reconciliation note (RESEARCH vs CONTEXT): D-11 named `globalTeardown` for the merge, but `globalTeardown` only receives `FullConfig` and cannot see `FullResult.status` or the planned `Suite`, so it cannot gate the prune (D-09). The merge therefore lives in the custom REPORTER (`onBegin` captures the completeness signal + planned suite; `onEnd` performs merge + the gated prune). The reporter stays summary-only for OUTPUT (the healed/could-not-heal rendering is untouched); the merge is a teardown-time side effect it performs because it is the only component holding both signals. The destructive prune is additionally gated behind opt-in env `SELFMEND_PRUNE` (RESEARCH Open Q1: `--last-failed`/single-file runs are not detectable in 1.60); refresh-on-pass (non-destructive) always runs on a green write.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker shards -> main-process merge | N untrusted shard files (possibly stale/foreign-version) are read and merged into the committed file by the single main-process writer. |
| FullConfig/FullResult -> prune decision | A destructive prune is gated on run-completeness signals; misreading them deletes valid baselines (Pitfall 2). |
| committed baseline.json -> git | The merged file is committed and shared; it must stay PII-free, deterministic, and never the product of an interrupted write. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | Tampering / DoS | reporter onEnd reading foreign/old shards | mitigate | readShards parses each via the Plan 01 safe loader and skips bad shards; the merge side effect is wrapped so a failure logs a warning, never crashing the user's run at teardown (Pitfall 5). |
| T-03-09 | Tampering (data loss) | destructive prune on an incomplete/filtered run | mitigate | prune gated on isComplete (match-all grep + null grepInvert + null shard) AND result.status==='passed' AND opt-in SELFMEND_PRUNE; refresh-on-pass is the only non-opt-in write. Proven by reporter gate test + prune.spec (D-09, Pitfall 2). |
| T-03-10 | Information Disclosure | committed baseline.json (PII) | mitigate | Entries are the Plan 01 schema's derived signals only (D-02); the phase gate inspects a real baseline to confirm no innerHTML/raw DOM; serializer is deterministic (D-03). |
| T-03-11 | Tampering (integrity) | interrupted teardown write corrupting baseline.json | mitigate | Single-writer atomicWrite (temp+rename+retry) from Plan 02; workers never touch baseline.json (CAP-03 anti-pattern avoided). |
| T-03-SC | Tampering | npm installs | mitigate | RESEARCH recommended path adds NO new runtime dependency (node:fs + existing zod); `write-file-atomic` is a documented fallback only and is NOT installed by this phase. No package-install task -> no legitimacy checkpoint required. |
</threat_model>
